4. 결과를 “눈에 보이게” 만들어야 함

경진대회는 보는 사람이 이해하기 쉬워야 한다.

상위권은 보통 결과가 시각적으로 깔끔하다.

추천 순서:

질문 입력
상단: 한 줄 요약
중단: 표/차트/그래프
하단: 출처 또는 디버그 요약

예를 들어 장애현황은:

상단: “오늘 장애 4건, 주요 원인은 DB 연결 실패와 파일 없음”
아래: 실제 장애 테이블

이용내역은:

상단: “최근 6개월 월별 이용금액 추이”
아래: bar chart
필요하면 raw data 펼치기

이렇게 하면 심사위원이 좋아한다.

상위권 만들려면 지금 뭘 더 해야 하냐
가장 중요한 보강 5개
1. 데모 시나리오 고정

아무 질문이나 받으려고 하지 말고,
잘 되는 대표 질문 5~7개를 준비해.

2. 질문별 결과 형태 명확화
업무 개요 → 텍스트
배치 프로세스 → 단계형 설명
배치 흐름도 → graphviz
테이블 리니지 → graphviz
장애현황 → 표 + 요약
이용내역 → 차트

이게 일관돼야 한다.

3. fallback 정리

예:

DB 연결 실패 → 친절한 문구
조회 결과 없음 → “오늘 장애는 없습니다”
LLM 실패 → 표는 유지, 요약만 생략

이런 게 있어야 안정적이다.

4. 발표 자료에 아키텍처 한 장

이 그림 하나 있으면 좋다:

사용자 질문
→ 질문 정규화/오타 보정
→ intent 분기
→ 문서형(RAG) / 구조형(JSON 시각화) / 실시간형(DB 조회)
→ UI 출력

이 그림 한 장이 점수 차이 난다.

경진대회 기준 평가

보수적으로 보면:

청킹 없음 → 마이너스 요소 맞음
하지만
문서 구조화
metadata filter
질문 정규화
실시간 DB 조회 분리
시각화 분리

이걸 잘 설명하면 전체를 무너뜨릴 정도는 아님

즉,
“RAG 기술 디테일”만 세게 보는 심사위원에겐 약점이고,
